## Run 001 Summary

Minimal drag-to-assign. Added `Act1IntentClassificationDraggableCard` (temporary drag preview, fade
the source card, disable source raycast during drag) and `Act1IntentClassificationDropTarget`
(detect a dropped card on an intent group and forward card id + intent id to the presenter). Added
`AssignCardToIntent` to the controller so drag assignment targets a specific card without toggling
selection. The presenter attaches draggable behaviours to rendered cards and drop targets to
rendered group areas.

## Run 002 Summary

Bidirectional drag + wider drop area. Opaque card-like drag preview (fade only the source
placeholder); `DropTarget` extended to support both intent-group targets and an unassigned target;
unassigned drop target attached to the left card list; intent drop targets attached to group panels
and their assigned-card scroll viewports (drops anywhere in the broader group area are accepted);
assigned rows made draggable — drag back to the left = unassign, drag to a different group =
reassign; a drop onto the card's current group is ignored (no same-group reorder); assigned rows
made more compact.

## Run 003 Summary

Drag visual cleanup + compact rows. Introduced a single static active drag source/preview;
`CancelActiveDrag()` clears any previous preview when a new drag starts; `CompleteDragVisuals()`
cleans the preview before assignment/unassignment callbacks that may re-render and destroy the source
row; `OnDisable()` clears the preview if the source row is destroyed during re-render; preview hidden
immediately before deferred `Destroy` to avoid one-frame leftovers. Drops stay group-based. Assigned
rows made more compact (shorter height, smaller text, tighter padding, soft outline/colour).

## Drag-to-Assign Behaviour

Dragging a left-side message card onto an intent group area assigns it via
`AssignCardToIntent` → `IntentClassificationSession.MoveCardToGroup`; the assigned-card list
re-renders to reflect the assignment.

## Bidirectional Drag Behaviour

Assigned rows are draggable: dragging an assigned card back to the left message-card list unassigns
it (via the controller/session unassign path); dragging it to a different group reassigns it;
dropping onto its current group is ignored.

## Group Drop-Zone Behaviour

Drops are group-based, not coordinate-based free placement — a drop anywhere within a group panel
(including its assigned-card scroll viewport) counts as assignment to that group.

## Drag Visual Cleanup Behaviour

Exactly one solid drag preview exists at a time; it is cleaned reliably on successful drop, failed
drop, a new drag starting, or the source row being destroyed during re-render — so no ghost/
afterimage artifacts remain.

## Compact Assigned-Row Behaviour

Right-side assigned cards render as compact, chip-like rows that stay readable inside the scrollable
group areas.

## Controller/Session Routing

All assign / unassign / reassign actions route through
`Act1IntentClassificationInteractionController` and `IntentClassificationSession`. The drag/drop
scripts are presentation-only; no pure puzzle logic was edited.
